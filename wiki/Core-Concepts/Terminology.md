# Terminology

Standard vocabulary used across SoloDev documentation and codebase. Contributors and users should use these terms consistently.

## Platform Terms

| Term | Definition |
|------|-----------|
| **SoloDev** | The open-source, AI-native DevOps platform. Built from a fork of Gitness by Harness. |
| **SoloDev Loop** | The core feedback cycle: Detect → Analyze → Propose → Validate → Apply. |
| **Space** | An organizational container (analogous to a GitHub organization). All modules are scoped to a space. |
| **Module** | A discrete functional unit of the platform (e.g., Error Tracker, Security Scanner, Quality Gates). |

## Architecture Terms

| Term | Definition |
|------|-----------|
| **Layer 1 — Core** | Components inherited from Gitness: SCM, Pipelines, Gitspaces, Registry. |
| **Layer 2 — Observability** | Signal-producing modules: Error Tracker, Security Scanner, Quality Gates, Health Monitor, Feature Flags, Tech Debt. |
| **Layer 3 — Intelligence** | AI-driven modules: AI Worker, Error Bridge, Solo Gate, Auto-Pipeline, MCP Server, Security Remediation. |
| **Layer 4 — UI / CLI** | Web Dashboard (React/TypeScript) and CLI entry points. |
| **Control Plane** | Orchestration components that coordinate pipelines, signals, and AI modules. |

## AI and Remediation Terms

| Term | Definition |
|------|-----------|
| **Remediation** | A structured task representing a detected failure and its proposed fix. Contains error context, AI analysis, patch diff, and confidence score. |
| **AI Worker** | Background service that polls pending remediation tasks, sends context to an LLM, and stores the resulting patch. |
| **Error Bridge** | Service that automatically creates remediation tasks when errors are reported or pipelines fail. |
| **Patch Diff** | A unified diff (`patch -p1` compatible) generated by the AI Worker as a proposed code fix. |
| **Confidence Score** | A 0.0–1.0 value estimating the reliability of an AI-generated patch. |
| **LLM Provider** | The external AI model used for patch generation. Supported: Anthropic, OpenAI, Google Gemini, Ollama. |

## Signal Terms

| Term | Definition |
|------|-----------|
| **Signal** | Any platform event indicating something requires attention: pipeline failure, runtime error, security finding, quality violation, health check failure. |
| **Error Group** | A collection of similar error occurrences, grouped by fingerprint (SHA256 of title + stack trace). |
| **Error Occurrence** | A single instance of a reported error, including stack trace, environment, and runtime metadata. |
| **Finding** | A security issue detected by the Security Scanner (vulnerability, secret, code smell, bug). |
| **Health Check** | A configured HTTP endpoint monitor that tracks uptime and response time. |

## Enforcement Terms

| Term | Definition |
|------|-----------|
| **Solo Gate** | The enforcement engine that applies quality and security policies adapted for solo developers. |
| **Enforcement Mode** | One of three modes: `strict` (block all findings), `balanced` (block critical/high, warn medium), `prototype` (never block, log as debt). |
| **Quality Rule** | A policy definition checked against code (e.g., `coverage >= 80`, `max_function_lines <= 50`). |
| **Quality Evaluation** | The result of checking a commit against all active quality rules. |
| **Tech Debt Item** | A tracked issue representing deferred work, linked to files and categorized by type. |

## Agent Terms

| Term | Definition |
|------|-----------|
| **MCP** | Model Context Protocol — the standard for AI agent tool access. SoloDev implements the MCP 2024-11-05 specification. |
| **Atomic Tool** | An MCP tool that wraps a single controller method (1:1 mapping). |
| **Compound Tool** | An MCP tool that orchestrates multiple controllers into a high-level workflow. |
| **Resource** | An MCP read-only data URI providing real-time platform state to AI agents. |
| **Prompt Template** | A pre-built MCP prompt encoding SoloDev domain expertise for common agent workflows. |

## Pipeline Terms

| Term | Definition |
|------|-----------|
| **Auto-Pipeline** | Module that detects the tech stack from file paths and generates CI/CD YAML without manual configuration. |
| **Pipeline Execution** | A single run of a CI/CD pipeline, tracked by execution number. |
| **Detected Stack** | The language, framework, build tool, and test presence inferred from repository file paths. |

## Status Labels

| Label | Meaning |
|-------|---------|
| **Concept** | Described in documentation but no implementation exists. |
| **Prototype** | Initial implementation exists but is not production-ready. |
| **Implemented** | Code compiles, runs, and exposes functional API endpoints. |
| **Working** | Synonym for Implemented in older documentation. |
| **Planned** | On the roadmap but not yet started. |
